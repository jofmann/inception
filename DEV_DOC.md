# Developer Documentation

## Prerequisites

The following tools must be installed on the host machine before setting up the project:

- **Docker** — container runtime
- **Docker Compose** — container orchestration (included with Docker Desktop, or install separately as `docker-compose-plugin`)
- **make** — build automation
- **openssl** — used to generate the self-signed SSL certificate (usually pre-installed)

On a Debian-based system:
```bash
sudo apt-get update
sudo apt-get install -y docker.io docker-compose-plugin make openssl
sudo usermod -aG docker $USER
# Log out and back in for the group change to take effect
```

## Environment Setup from Scratch

### 1. Clone the repository

```bash
git clone <repository-url>
cd inception
```

### 2. Create the `.env` file

Copy the example and fill in your values:

```bash
cp srcs/.env.example srcs/.env
```

Open `srcs/.env` and set the following variables:

```env
DOMAIN_NAME=<your-login>.42.fr

MYSQL_DATABASE=inception_db
MYSQL_USER=maria

WORDPRESS_URL=https://<your-login>.42.fr
WORDPRESS_TITLE=Inception
WORDPRESS_DB_HOST=maria-container
WORDPRESS_ADMIN_USER=<your-admin-username>
WORDPRESS_ADMIN_EMAIL=<your-email>
WORDPRESS_USER=<regular-user>
WORDPRESS_USER_EMAIL=<regular-user-email>

FTP_USER=ftpuser
```

### 3. Add the domain to `/etc/hosts`

```bash
echo "127.0.0.1 <your-login>.42.fr" | sudo tee -a /etc/hosts
```

### 4. Run make

```bash
make
```

`make` will automatically:
- Generate random passwords for all services and store them in `secrets/`
- Generate a self-signed SSL certificate for your domain
- Create the host volume directories under `~/data/`
- Build all Docker images
- Start all containers in the background

## Makefile Targets

| Target | Description |
|--------|-------------|
| `make` / `make up` | Generate secrets if missing, then build and start all containers |
| `make build` | Build all Docker images without starting containers |
| `make down` | Stop and remove containers and the network |
| `make stop` | Stop containers without removing them |
| `make clean` | Stop containers and delete all volume data on the host |
| `make fclean` | Full cleanup — removes containers, images, volumes, and cached layers |
| `make re` | `fclean` followed by `all` — full rebuild from scratch |

## Docker Compose

The Compose file is located at `srcs/docker-compose.yml`. To run Compose commands directly:

```bash
# Start in foreground (see all logs)
docker compose -f srcs/docker-compose.yml up

# Rebuild a single service without touching the others
docker compose -f srcs/docker-compose.yml build mariadb
docker compose -f srcs/docker-compose.yml up -d --no-deps mariadb

# View running services
docker compose -f srcs/docker-compose.yml ps
```

## Managing Containers

**List all running containers:**
```bash
docker ps
```

**Open a shell inside a container:**
```bash
docker exec -it maria-container bash
docker exec -it wordpress-container bash
docker exec -it nginx-container bash
docker exec -it ftp-container sh        # Alpine uses sh, not bash
```

**View logs:**
```bash
docker logs maria-container
docker logs -f wordpress-container      # follow in real time
```

**Inspect a container (network, mounts, env):**
```bash
docker inspect maria-container
```

**Check MariaDB health:**
```bash
docker inspect --format='{{.State.Health.Status}}' maria-container
# healthy = MariaDB is accepting connections
```

**Monitor container events in real time:**
```bash
docker events --filter container=maria-container
```

```bash
docker compose -f srcs/docker-compose.yml down --volumes
```

## Where Data Is Stored

All persistent data lives on the host under `~/data/`:

| Data | Host path | Container path |
|------|-----------|----------------|
| MariaDB database | `~/data/db/` | `/var/lib/mysql` |
| WordPress files | `~/data/wordpress/` | `/var/www/html` |

These paths are bind-mounted into named Docker volumes defined in `docker-compose.yml`:

```yaml
volumes:
  mariadb-data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/<login>/data/db
  wordpress-data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/<login>/data/wordpress
```

Data persists across `docker compose down` and `docker compose up` because it lives on the host filesystem. It is only deleted when you run `make clean` or `make fclean`.

**Secrets** are stored in `secrets/` at the repository root and mounted into containers as Docker Secrets at `/run/secrets/<name>`. They are never written to the container filesystem — they are mounted as `tmpfs` in memory only.

## Rebuilding After Code Changes

When you change a Dockerfile or an init script, Docker's build cache will detect the change and rebuild only the affected layers. You do not need to run `make fclean`.

To rebuild a single service quickly:
```bash
docker compose -f srcs/docker-compose.yml build <service>
docker compose -f srcs/docker-compose.yml up -d --no-deps <service>
```

Replace `<service>` with `mariadb`, `wordpress`, `nginx`, or `ftp`.

To force a full rebuild ignoring the cache:
```bash
docker compose -f srcs/docker-compose.yml build --no-cache <service>
```
